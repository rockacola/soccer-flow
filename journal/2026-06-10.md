## Debugging / troubleshooting

### Problem 1 — `assertVisible: "Test Team"` fails after Save

**Symptom:** Maestro flow completes team creation (tap Save on the modal), then immediately fails the assertion `assertVisible: "Test Team"`. Failure reproduced twice: once at end of the 2026-06-09 session and again at start of 2026-06-10.

**First hypothesis:** The team name `<Text>` is inside a `Swipeable` component from `react-native-gesture-handler`. Suspected the gesture wrapper interferes with Maestro's accessibility tree traversal, making the inner text invisible to it.

**Investigation:** Read `TeamRow.tsx`. Structure is:

```
<Swipeable>
  <TouchableOpacity>
    <View>          ← colourDot
    <View>          ← rowText
      <Text>        ← team name lives here
      <Text>        ← player count
    </View>
    <Text>          ← chevron ›
  </TouchableOpacity>
</Swipeable>
```

The name is a plain `<Text>` with no `accessibilityLabel` set. `Swipeable` is a layout wrapper — it shouldn't block Maestro from seeing child text nodes.

**Second hypothesis (correct):** Timing issue. The modal uses `animationType="slide"` (`presentationStyle="pageSheet"`). After tapping Save, the sheet animates down. The original fix (`waitForAnimationToEnd`) is a one-shot check at the moment Maestro decides the animation has ended — but that moment is not guaranteed to coincide with the sheet being fully off-screen and the list being settled in the view hierarchy.

**Fix:** Replace `waitForAnimationToEnd` + `assertVisible` with `extendedWaitUntil`:

```yaml
- extendedWaitUntil:
    visible:
      text: 'Test Team'
    timeout: 5000
```

`extendedWaitUntil` polls repeatedly until the condition is true or the timeout expires, which handles any residual animation delay without relying on a single synchronisation point.

### Problem 2 — Recording saved to `~/.maestro/tests/…`, not the repo

**Symptom:** `maestro record --local` writes the video to `~/.maestro/tests/<timestamp>/maestro-recording.mp4`. This path changes every run, making it hard to find and not part of the repo.

**Fix:** Add `--output` flag to pin the destination:

```
maestro record --local --output .maestro/recordings/create_team.mp4 .maestro/create_team.yaml
```

Also:

- Created `.maestro/recordings/` with `.gitkeep` so the directory is always present when the script runs
- `*.mp4` is already gitignored so the recording itself is never committed

### Problem 3 — `assertVisible: "James"` fails (and Marcus, Oliver, Noah)

Same root cause as Problem 1 — `PlayerRow` also wraps `TouchableOpacity` in `Swipeable`. Applied the same fix: `accessibilityLabel={player.name}` on the `TouchableOpacity` in `PlayerRow.tsx`.

### Problem 4 — `--output` flag not recognised

`maestro record` has no `--output` flag. The output path is a positional argument after the flow file:

```
maestro record --local <flowFile> [<outputFile>]
```

### Problem 5 — Recording resolution is low

No resolution flag exists in `maestro record`. Recording quality is determined by the simulator device's screen resolution. Higher-density devices (e.g. iPhone 16 Pro Max) give more pixels. Nothing configurable in the script.

## Decisions made

- `extendedWaitUntil` is the right primitive for any assertion that follows a modal dismiss or screen transition; `waitForAnimationToEnd` is too coarse.
- Any `TouchableOpacity` inside a `Swipeable` needs an explicit `accessibilityLabel` — the gesture wrapper interferes with Maestro's accessibility tree traversal. Applied to `TeamRow` and `PlayerRow`; `MatchRow` will need the same when a flow covers it.
- `ios/` gitignored — it's a generated directory (`expo run:ios`), not worth committing.
- Recordings live at `recordings/<timestamp>_<flow>.mp4` at the project root — timestamped for chronological ordering, gitignored via `recordings/*.mp4`, directory tracked via `.gitkeep`.
- Maestro has no `pause` or `sleep` command. Waiting a fixed duration requires `evalScript: ${new Promise(resolve => setTimeout(resolve, <ms>))}`.

### `match_lifecycle` flow — attempted and removed

Attempted a full match lifecycle Maestro flow (create team → start match → record goal + note → wait for Half Time / 2nd Period → finish → assert match record). Ran into two blockers:

1. **Phase label not findable by text.** Maestro does exact text matching. The period info `TouchableOpacity` aggregates its children into one accessibility element (`"1st Period | 7:55 am – 7:56 am, 00:27"`), so `assertVisible: "1st Period"` fails. Fix is `accessibilityLabel` on the `TouchableOpacity` — works, but adds production code noise for a flow that was cut.

2. **`RemarkModal` keyboard blocks "Record Note" button.** The modal has no `KeyboardAvoidingView`, so the software keyboard covers the submit button. Maestro's `hideKeyboard` does not work with multiline `TextInput`. Restructuring with `KeyboardAvoidingView` did not fully resolve it.

**Decision:** Remove the flow and revert the code changes. The `RemarkModal` keyboard issue is a real UX bug — address it separately as a focused fix.
