Added `ScreenBackground` (ImageBackground wrapper) to all screens and modals. Added `expo-blur` `BlurView` to list rows (TeamRow, PlayerRow, MatchRow) for a frosted-glass effect over the background image. Introduced per-section background images via a `variant` prop on `ScreenBackground` — match screens use `bg-match`, settings uses `bg-settings`, team screens use the default.

## Decisions made

`BlurView` with `StyleSheet.absoluteFill` placed behind row content rather than wrapping it — keeps layout in `Pressable`, blur is purely cosmetic. `overflow: hidden` on the row clips the blur to its bounds. Row press feedback dropped in favour of a static blur (no opacity change on press). `variant` prop on `ScreenBackground` keeps image asset references out of screen files — all four sources pre-required in one place.

## Problems / friction

Metro bundler error when `require()` paths included `@3x` suffix — Metro resolves density variants automatically from the base name, so `bg-match.png` resolves to `bg-match@3x.png` on device.

## What I worked on (session 2)

Completed full accessibility label audit across all screens and modals. Added and fixed the `create_team` Maestro flow to handle the onboarding screen. Expanded unit test coverage for all utils and services.

## Decisions made

Rule established: only add `accessibilityLabel` when visible text genuinely can't stand alone — icon/symbol buttons, disabled Views, or inputs with no visible label. Remove any label that differs from (or paraphrases) visible text, since Maestro uses the accessibility tree and a mismatched label breaks automation selectors. Visible button text is the source of truth for both VoiceOver and Maestro.

Changed "Finish" button text to "Finish Match" to make it self-describing, then dropped the explicit label.

## What I learned

React Native inherits child `Text` content as the accessibility label automatically when no explicit label is set on `TouchableOpacity`. Setting an explicit label overrides this — causing Maestro to see a different string than what's visually shown.

## Problems / friction

`AddPlayerModal.tsx` edit failed with "file modified since read" (linter reformatted on save). Required re-reading before each edit attempt.

## What I worked on (session 3)

Added `manage_team` Maestro flow: loads sample data, renames a team and changes its colour, edits a player's jersey number and position. Fixed `PlayerRow` accessibility so Maestro can assert individual child elements (name, jersey, position).

## Decisions made

`Pressable` with `accessible={false}` makes the row a transparent accessibility container — child `Text` nodes are individually queryable by Maestro while `onPress` still fires normally. This is the right model for list rows where you want both a single tap target and individually assertable child content. Contrast with `TouchableOpacity` which inherits child text as its label automatically.

Maestro assertions after an edit should target a value that is unique and specific to the change made — jersey "22" is unambiguous proof the edit worked; "MID" alone would match any mid-field player.

## What I learned

`Pressable` does not inherit child text as its accessibility label the way `TouchableOpacity` does. Without an explicit `accessibilityLabel`, a `Pressable` with `accessibilityRole="button"` has no label at all — Maestro cannot find it. `accessible={false}` is the escape hatch to expose children individually. Maestro does not support `clearText` or `replaceText`; the correct command is `eraseText: N`.
