## What I learned

React Navigation's back caret only appears when there's a previous screen in the current stack. Cross-tab `navigate()` doesn't reliably create that history, so screens navigated to cross-tab need an explicit `headerLeft` if a back affordance is required.

`navigate('ScreenName')` in a stack can push a new instance rather than popping to an existing one — always prefer `popToTop()` or `goBack()` when the intent is to go back, not forward.

## Problems / friction

Swipe-to-delete UX in React Native is not a platform primitive. Getting it to feel native (physics, haptics, animation curve) requires significant additional work beyond what `react-native-gesture-handler`'s `Swipeable` provides out of the box.

## Next step

UI consideration: replace swipe-to-delete with an explicit delete button inside detail screens, then revisit gesture-based deletion as a polish pass once the feature set is stable.

## Maestro setup (later session)

## What I worked on

Set up Maestro for UI journey recording. Installed Maestro CLI, resolved Java dependency, wrote the first flow (`create_team.yaml`), and hit an iOS Simulator runtime blocker.

## Decisions made

- Added `accessibilityLabel` to colour swatches in `CreateTeamModal` so Maestro can target them by name (e.g. `"Blue"`) rather than coordinates — also improves accessibility for screen readers.
- Opted out of Maestro anonymous analytics globally via `MAESTRO_CLI_NO_ANALYTICS=1` in `~/.zshrc`.
- Flows live in `.maestro/`, recordings (`.mp4`) are gitignored.
- One npm script per flow: `maestro:create_team` → `maestro record .maestro/create_team.yaml`.

## What I learned

Maestro is a JVM-based tool — it requires Java 11+ and will silently fail with "command not found" or a JRE error without it. Install via `brew install --cask temurin@21` (needs sudo password interactively). The Maestro install script handles PATH in `~/.zshrc` but requires a shell reload (`source ~/.zshrc`) to take effect.

`xcrun simctl list runtimes` returning empty means iOS simulator runtimes are not downloaded even if Xcode is installed. Runtimes must be downloaded separately via Xcode → Settings → Platforms, or `xcodebuild -downloadPlatform iOS`.

## Problems / friction

- Maestro install script adds to PATH but the current terminal session doesn't pick it up — `maestro` appears broken until the shell is reloaded.
- Xcode 26.3 ships with no simulator runtimes pre-installed; `npm run ios` fails with "No iOS devices available" until a runtime is downloaded (several GB).
- Colour swatches had no text or accessibility labels, making them untargetable by Maestro without coordinate hacks.

## Steps taken (in order)

1. Ran `curl -Ls "https://get.maestro.mobile.dev" | bash` — Maestro CLI installed to `~/.maestro/bin`
2. Shell profile updated automatically (`~/.zshrc`, `~/.bash_profile`) but current session needed `source ~/.zshrc` to pick up the new PATH
3. `maestro --version` failed: "Unable to locate a Java Runtime" — Maestro requires JDK 11+
4. Installed Java via `brew install --cask temurin@21` (prompted for sudo password in terminal)
5. Verified: `java -version` → OpenJDK 21.0.11, `maestro --version` → 2.6.0
6. Added `export MAESTRO_CLI_NO_ANALYTICS=1` to `~/.zshrc` to opt out of analytics globally
7. Created `.maestro/create_team.yaml` flow and `maestro:create_team` npm script
8. Ran `npm run ios` → failed: "No iOS devices available in Simulator.app"
9. Ran `xcrun simctl list runtimes` → empty; no simulator runtimes installed despite Xcode 26.3 being present
10. Ran `open -a Simulator` → no devices available
11. Ran `xcodebuild -downloadPlatform iOS` → downloading iOS 26.3.1 Simulator (arm64), 8.39 GB — **in progress**

## Steps taken (continued)

12. iOS 26.3.1 Simulator runtime downloaded successfully — iPhone 17 Pro, iPhone Air, iPad Pro and others now listed in `xcrun simctl list devices available`
13. Ran `npm run ios` → new error: `osascript -e tell app "System Events" to count processes` exited with code 1
14. Confirmed with `osascript -e 'tell app "System Events" to count processes'` → "Not authorised to send Apple events to System Events (-1743)"
15. Fixed by going to **System Settings → Privacy & Security → Automation** → enabled **System Events** for **iTerm**
16. `npm run ios` succeeded → app opened in Expo Go on iPhone 17 Pro (iOS 26.3)
17. Ran `npm run maestro:create_team` → Maestro connected to the device but immediately crashed:
    `SimctlError: Failed to get app binary directory for bundle com.travaganza.soccerflow`

**Root cause of step 17:** `npm run ios` runs the app inside **Expo Go** (`host.exp.Exponent`), not as a standalone native binary. Maestro looks for the app by its own bundle ID (`com.travaganza.soccerflow`) which doesn't exist on the simulator — only Expo Go does.

**Fix:** use `npx expo run:ios` instead, which compiles and installs the app natively on the simulator with the correct bundle ID.

## Next step

- Re-run `npm run maestro:create_team` — Maestro should now find `com.travaganza.soccerflow` and execute the flow
